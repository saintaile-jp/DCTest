# Dreamcastゲームプロジェクト構成・開発ガイド

この文書は、このプロジェクトに用意したディレクトリとファイルの役割、構成を決めた理由、DreamSDKとKallistiOSを使ってゲームを作る手順を説明するものです。

## 1. プロジェクトの目的

このプロジェクトは、Dreamcast向けゲーム開発をすぐ開始できる最小構成です。現在のプログラムは、次の基本要素を含んでいます。

- KallistiOSによるDreamcastハードウェアの初期化
- PowerVRを使った画面描画
- Dreamcastコントローラーの入力取得
- 更新と描画を繰り返すゲームループ
- ELFへのリソース組み込み
- 実機転送用ELFとセルフブートCDIの生成

画面中央の四角を十字キーまたはアナログスティックで動かし、STARTボタンで終了できます。この小さなプログラムを土台に、画像、マップ、敵、サウンドなどを追加していく想定です。

## 2. ディレクトリ構成

```text
DCTest/
|-- src/               ゲームのCソースコード
|   `-- main.c
|-- romdisk/           ELF内へ埋め込む小さなリソース
|   `-- readme.txt
|-- disc/              CDIのデータトラックへ収録するファイル
|   |-- .gitkeep
|   `-- 1ST_READ.BIN   ビルド時に生成
|-- build/             コンパイル結果と配布用イメージ
|   |-- .gitkeep
|   |-- main.o         ビルド時に生成
|   |-- dctest.elf     実機転送・デバッグ用
|   |-- dctest.bin     CDI作成の中間ファイル
|   |-- IP.BIN         ブートセクター
|   `-- dctest.cdi     エミュレーター・CD-R用
|-- Makefile           KallistiOS用ビルド定義
|-- build.ps1          PowerShell用ビルドラッパー
|-- ip.txt             IP.BINへ格納するタイトル情報
|-- .gitignore         自動生成物のGit除外設定
|-- ReadMe.md          短いクイックスタート
`-- PROJECT_GUIDE.md   この詳細ガイド
```

### `src/`

ゲームのソースコードを置く場所です。現在はエントリーポイントを含む `main.c` だけですが、ゲームが大きくなったら、例えば次のように分割します。

```text
src/
|-- main.c
|-- game.c
|-- game.h
|-- player.c
|-- player.h
|-- input.c
|-- input.h
|-- graphics.c
`-- graphics.h
```

ソースを追加した場合は、`Makefile` の `SOURCES` に追加します。

### `romdisk/`

実行ファイルであるELFへ直接組み込むリソースを置きます。KallistiOSによって実行時に `/rd` としてマウントされるため、`romdisk/readme.txt` はゲーム内から `/rd/readme.txt` というパスで開けます。

設定ファイル、小さな画像、短い効果音など、常にすぐ参照したい小規模なデータに向いています。すべてELFの容量とメモリ使用量に影響するため、大容量の音楽やムービーを無制限に置く用途には向きません。

### `disc/`

CDIのデータトラックへ収録するファイルを置きます。大きなステージデータ、BGM、ムービーなど、ディスクから必要に応じて読むデータに適しています。

`1ST_READ.BIN` はDreamcastが起動時に実行するスクランブル済みプログラムで、`make disc` によって自動生成されます。通常は直接編集しません。

### `build/`

オブジェクトファイル、ELF、BIN、IP.BIN、CDIなどの生成物をまとめる場所です。ソースではないため、`.gitkeep` を除いてGitの管理対象から外しています。

主な生成物は次のとおりです。

| ファイル | 用途 |
|---|---|
| `main.o` | `main.c`をDreamcast用SH-4コードへコンパイルした中間ファイル |
| `dctest.elf` | デバッグ情報を持つKallistiOS実行ファイル。`dc-tool`での転送にも使用 |
| `dctest.bin` | ELFから変換した生の実行バイナリ |
| `IP.BIN` | Dreamcastディスクのブート情報 |
| `dctest.cdi` | エミュレーターまたはCD-Rで使用するセルフブートイメージ |

## 3. 各ファイルの解説

### `src/main.c`

現在のゲーム本体です。処理は大きく次の部分に分かれています。

1. `KOS_INIT_FLAGS(INIT_DEFAULT)`でKallistiOSの標準機能を初期化する。
2. `initialize_graphics()`でPowerVRを初期化し、単色ポリゴンの描画設定を作る。
3. `update()`でコントローラーを読み、プレイヤー座標を更新する。
4. `render()`で外枠とプレイヤーの四角を描画する。
5. `while(game.running)`で更新と描画を毎フレーム繰り返す。
6. STARTが押されたらループを抜け、PowerVRを終了する。

DreamcastのPowerVRでは、四角形も頂点の並びとして送信します。`draw_rectangle()`は、この頂点データ作成を一か所へまとめるための関数です。

### `Makefile`

DreamSDK Shell内でGNU Makeが読むビルド定義です。KallistiOS共通ルールである `$(KOS_BASE)/Makefile.rules` を利用しています。

主要ターゲットは次のとおりです。

| コマンド | 処理 |
|---|---|
| `make` | ソースとromdiskをコンパイルし、`dctest.elf`を生成 |
| `make run` | ELFをビルドし、設定済みの`KOS_LOADER`で実機へ転送 |
| `make disc` | `1ST_READ.BIN`、`IP.BIN`、セルフブートCDIを生成 |
| `make dist` | ELFからデバッグシンボルを除去 |
| `make clean` | 自動生成されたビルド成果物を削除 |

CDI作成では、ELFからBINへの変換、Dreamcast用スクランブル、IP.BIN生成、`makedisc`によるイメージ化を順番に行います。

### `build.ps1`

通常のPowerShellからDreamSDK Runnerを呼び出すための補助スクリプトです。DreamSDKを `C:\DreamSDK` へインストールしていることを前提としています。

```powershell
.\build.ps1
.\build.ps1 -Target clean
.\build.ps1 -Target run
.\build.ps1 -Target disc
```

DreamSDKの場所を変更した場合は、スクリプト内の `$runner` を実際のパスに合わせます。

### `ip.txt`

Dreamcastディスクのブートセクター `IP.BIN` に入るメタデータです。ゲーム名、製作者名、対応地域、起動ファイル名などを定義します。

公開するゲームにするときは、少なくとも `Product No`、`Version`、`SW Maker Name`、`Game Title` を作品に合わせて変更します。`Game Title`などの各フィールドには長さや使用可能文字の制限があるため、DreamSDK付属の`makeip`資料も確認してください。

### `.gitignore`

ELF、BIN、CDI、オブジェクトファイルなど、ソースから再生成できるファイルをGitへ登録しないための設定です。リポジトリには原則としてソース、設定、手作業で作ったアセットを保存します。

### `.gitkeep`

Gitは空ディレクトリを管理しないため、`build/`と`disc/`の存在を保つ目的で置いた空ファイルです。ゲームから読み込むデータではありません。

## 4. この構成を用意した手順と設計理由

構成は次の順序で用意しました。

1. インストール済みDreamSDKとKallistiOSの公式サンプルを確認した。
2. DreamSDKの環境変数と `Makefile.rules` に従うMakefileを作成した。
3. ソース、組み込みリソース、ディスク収録データ、生成物を別ディレクトリに分離した。
4. 入力、状態更新、描画を含む最小ゲームループを実装した。
5. romdiskをELFへリンクする設定を追加した。
6. 実機転送用ELFだけでなく、セルフブートCDIまで生成できるルールを追加した。
7. DreamSDKの実ツールチェーンでELFをコンパイルした。
8. `elf2bin`、`scramble`、`makeip`、`makedisc`を実行し、CDI生成まで成功することを確認した。

ソースと生成物を分けたのは、開発中に編集すべきものを明確にし、不要なバイナリをGitへ混ぜないためです。また、`romdisk/`と`disc/`を分けたのは、常駐させたい小さなデータと、ディスクから読む大きなデータでは適切な扱い方が異なるためです。

最初のプログラムを単なる文字表示ではなく「入力で四角を動かすゲームループ」にしたのは、ゲーム制作で必須となる入力、更新、描画、終了処理を最初から確認できるようにするためです。

## 5. ゲームを作成していく手順

### 手順1: 現在の雛形を動かす

まずDreamSDK Shellでプロジェクトへ移動し、ビルドします。

```sh
cd /c/DreamSDK/MyProject/DCTest
make
```

PowerShellから実行する場合:

```powershell
Set-Location C:\DreamSDK\MyProject\DCTest
.\build.ps1
```

エミュレーターでは `build/dctest.cdi` を作成して読み込みます。

```powershell
.\build.ps1 -Target disc
```

### 手順2: ゲームの状態を定義する

プレイヤー座標だけでなく、スコア、残機、敵、弾、現在の画面などを構造体へ追加します。小規模なうちは `game_state_t` を拡張し、大きくなったらプレイヤーや敵を別の構造体とソースへ分割します。

### 手順3: 入力処理を追加する

`update()`でA、B、X、Y、トリガーなどを判定し、攻撃、ジャンプ、メニュー操作を実装します。ボタンを押している間の処理と、押した瞬間だけの処理を区別するには、前フレームのボタン状態を保存して比較します。

### 手順4: ゲームルールを実装する

移動制限、衝突判定、敵AI、得点、クリア・ゲームオーバー条件などを追加します。描画処理の中でゲーム状態を変更せず、原則として更新は `update()`、表示は `render()`へ分けると管理しやすくなります。

### 手順5: 画像を表示する

画像はDreamcastのPowerVRが扱えるテクスチャ形式へ変換してから読み込み、テクスチャ付きポリゴンとして描画します。小さな画像は `romdisk/`、大きな画像やステージごとの素材は `disc/`へ置きます。

PNGをゲーム内で直接読みたい場合は、DreamSDK Managerで対応するKOS Portを構築・インストールし、Makefileへ必要なライブラリを追加します。実行速度とメモリを重視する場合は、開発PC上で事前変換したDreamcast向けテクスチャを使う方法が適しています。

### 手順6: サウンドを追加する

短い効果音と長いBGMを分けて考えます。効果音はメモリへ読み込んで即時再生し、長いBGMはディスクからストリーミングする設計が基本です。Ogg、ADXなどを使う場合は、必要なKOS Portや再生ライブラリをDreamSDK Managerで準備します。

### 手順7: 実機とエミュレーターで検証する

エミュレーターは反復テストに便利ですが、描画タイミング、メモリ、コントローラー、GD-ROM相当の読み込み速度などは実機と完全には一致しません。節目ごとにDreamcast実機でも確認します。

実機へELFを転送するにはCoder's Cable、LAN Adapter、Broadband Adapterなどと、DreamSDK側の`dc-tool`設定が必要です。準備済みなら次を使います。

```sh
make run
```

### 手順8: 配布用イメージを作る

ゲームタイトルや製作者名を `ip.txt` に設定し、必要な外部データを `disc/`へ入れてCDIを生成します。

```sh
make clean
make disc
```

完成した `build/dctest.cdi` をエミュレーターで起動確認し、実際にCD-Rへ書き込む場合は対応するDreamcast本体と適切な書き込み方法を使用します。

## 6. 次に行うとよい拡張

1. `main.c`を入力、描画、ゲームロジックのファイルへ分割する。
2. 固定フレーム依存を避けるため、経過時間を使った更新を導入する。
3. タイトル、プレイ、ポーズ、結果画面の状態管理を追加する。
4. テクスチャ読み込みとスプライト描画機能を作る。
5. 当たり判定、敵、スコア、サウンドを順に追加する。
6. デバッグ表示とエラー処理を整備する。
7. 定期的に実機上の速度とメモリ使用量を確認する。

一度に大きなゲームへ作り替えるより、「入力できる」「1枚描画できる」「当たり判定が動く」「効果音が鳴る」という小さな完成状態を積み重ねると、Dreamcast固有の問題を切り分けやすくなります。
